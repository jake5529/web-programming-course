LR11: DevOps - Docker & CI/CD

Титульный

Цель: воспроизводимый локальный цикл build, run, test и release для backend.

Формат LR11: local-first, без удаленного деплоя.

План лекции

  1. Docker основы
  2. Dockerfile для Node.js backend
  3. Docker Compose для backend + database
  4. CI pipeline в GitHub Actions
  5. Local CD-like сценарий и rollback
  6. Практические checkpoints

Формат команд: ЛИНУКС (эталон) + WINDOWS (PowerShell).

Почему DevOps нужен уже сейчас

Без стандартизированного окружения качество кода сложно подтверждать командно.
  • Окружения отличаются
  • "У меня работает" не масштабируется
  • Проверки качества не автоматизированы

Docker в одной схеме

  • Dockerfile: рецепт сборки
  • Image: versioned immutable артефакт
  • Container: runtime-процесс

Один image должен одинаково запускаться у всех участников команды.

Image vs Container

  • Image: шаблон, версия, артефакт
  • Container: запущенный процесс с состоянием
Собираем image один раз, запускаем контейнеры многократно.

Dockerfile поток

  1. Выбрать базовый образ
  2. Установить зависимости
  3. Скопировать код
  4. Собрать TypeScript
  5. Запустить runtime

Порядок шагов влияет на кеш и скорость сборки.

Multi-stage build

  • builder stage: install + build
  • runner stage: только runtime
  • Меньше размер image
  • Меньше лишних зависимостей
  • Быстрее pull и запуск

`.dockerignore`

node_modules
.git
coverage
dist
.env

Сокращает build context, ускоряет сборку и снижает риск утечек.

Docker Compose роль

  • Поднимает backend + db одной командой
  • Создает общую сеть сервисов
  • Управляет lifecycle окружения
docker compose up --build -d
docker compose ps

Healthcheck и старт зависимостей

Проблема: backend стартует до готовности БД.

  • Добавляем healthcheck для db
  • Используем depends_on.condition: service_healthy
Меньше случайных падений при старте стека.

Environment variables

  • DATABASE_URL
  • JWT_SECRET
  • NODE_ENV
  • PORT

`.env` не коммитим, `.env.example` коммитим.

CI/CD в LR11

  • CI: lint, test, build, docker build
  • CD: локальная симуляция release + rollback
Удаленный деплой в этой лабораторной не выполняется.

GitHub Actions структура

  • on: triggers
  • jobs: задачи
  • steps: последовательность команд

Шаги располагаем от быстрых проверок к более тяжелым.

Пример CI workflow

name: CI
on: [push, pull_request, workflow_dispatch]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npm ci
      - run: npm run lint
      - run: npm run test
      - run: npm run build
      - run: docker build -t quiz-backend:ci .

Локальный запуск CI

  • act -W .github/workflows/ci.yml (опционально)
  • или локальный fallback-скрипт с тем же порядком шагов

Linux команды показываем как эталон, PowerShell как эквивалент для студентов на Windows.

npm ci
npm run lint
npm run test
npm run build
docker build -t quiz-backend:ci .

Local release сценарий

  1. Собрать image с новым tag
  2. Поднять compose
  3. Выполнить healthcheck
  4. Проверить 1-2 критичных endpoint
  • ЛИНУКС (эталон): export TAG=$(date +%Y%m%d-%H%M%S)
  • WINDOWS (PowerShell): $TAG = Get-Date -Format "yyyyMMdd-HHmmss"

Это минимальный CD-like процесс для локальной среды.

Rollback локально

  1. Остановить текущий стек
  2. Вернуть предыдущий image tag
  3. Запустить снова
  4. Повторить smoke-check
Rollback должен быть быстрым и воспроизводимым.

Частые ошибки

  • localhost вместо db в compose сети
  • Нет healthcheck
  • Секреты зашиты в Dockerfile
  • Нет tag-стратегии для rollback
  • CI шаги не совпадают с локальными

Финальные правила

  1. Окружение должно запускаться одной командой
  2. CI должен останавливать плохие изменения
  3. Release должен проверяться smoke-тестами
  4. Rollback должен быть готов заранее
  5. LR11 остается local-first без remote deploy
